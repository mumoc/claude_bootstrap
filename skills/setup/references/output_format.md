# Output Format Reference

Loaded by the `setup` skill during the generation phase.

---

## File layout

```
context/
  domain_glossary.md          ← shared, loaded by all agents on every run
  {service}_context.md        ← one per major service namespace
  .setup_manifest.json        ← machine-readable setup state (not loaded by agents)
```

For single-service repos, use `app_context.md` instead of per-service files.

---

## domain_glossary.md

**Purpose:** Shared vocabulary and system-wide rules. Every agent reads this first,
regardless of which service is involved.

**Structure:**

```markdown
# Domain Glossary
<!-- Generated by setup skill. Do not edit manually. Re-run setup to update. -->
<!-- Generated: {ISO timestamp} | Hash: {structural_hash} -->

## System Purpose

{One sentence: what problem this system solves, in business terms.}

## Primary Actors

| Actor | Role |
|---|---|
| {actor} | {what they do in this system} |

## Core Entities

| Entity | Business meaning |
|---|---|
| {entity} | {what it represents in business terms, not technical terms} |

## Domain Terminology

| Term | Meaning in this system |
|---|---|
| {term} | {definition — note if it differs from common usage} |

## System-Wide Rules

Rules that apply across all services:

- {rule}
- {rule}

## Known Constraints and Historical Decisions

- {constraint or decision}
- {constraint or decision}
```

**Rules:**
- Entity descriptions are business meanings, not schema descriptions.
- Never include column names, types, or database details.
- System-wide rules are rules that no single service owns — they span the whole domain.
- Keep under 500 tokens total.

## {service}_context.md

**Purpose:** Service-specific domain knowledge for the Analyzer, Challenger, and Planner
agents when a ticket touches that service.

**Naming:** Match the service namespace directory name.
Examples: `contracts_context.md`, `billing_context.md`, `fleet_context.md`.

**Structure:**

```markdown
# {Service Name} Context
<!-- Generated by setup skill. Do not edit manually. Re-run setup to update. -->
<!-- Generated: {ISO timestamp} -->

## What This Service Owns

{One paragraph: the domain concept this service is responsible for. What problem
does it solve? What does it NOT own (boundary statement)?}

## Key Objects

| Object | Role |
|---|---|
| {class or concept} | {one-line business role} |

## Business Rules

Rules that govern this service's behavior. Ordered by importance.

- {rule — use plain language, not code references}
- {rule}

## Lifecycle and States (if applicable)

{Entity name} lifecycle:

```
{state} → {state} → {state}
```

Transition rules:
- {what triggers transition}
- {what is forbidden}

## Known Constraints and Edge Cases

- {constraint}
- {historical decision or legacy rule}

## Integration Points

Services or systems this service depends on or is depended on by:

| Dependency | Direction | What crosses the boundary |
|---|---|---|
| {service} | inbound / outbound | {what data or event} |

## Gaps

{If any business rules could not be determined, list them here.}

- [NEEDS INPUT: {what is unknown}]
```

**Rules:**
- Business rules section is the most important — never skip it even if sparse.
- Mark every unknown with `[NEEDS INPUT: ...]` rather than leaving blank or guessing.
- Integration points only include meaningful domain dependencies, not technical ones
  (e.g. "depends on billing for invoice creation" not "calls BillingService.call").
- Keep under 600 tokens per file. Split into two files if a service is too large.

## .setup_manifest.json

Machine-readable record of the setup run. Not loaded by agents.

```json
{
  "schema_version": "1.0",
  "skill_version": "1.0",
  "completed_at": "<ISO 8601 timestamp>",
  "structural_hash": "<models>:<services>:<top_dirs>",
  "files_produced": [
    "context/domain_glossary.md",
    "context/contracts_context.md"
  ],
  "harvest_sources": [
    { "path": "<relative path>", "type": "namespace_readme | schema | claude_md | docblock" }
  ],
  "interview_conducted": true,
  "interview_gaps_filled": ["system_purpose", "contracts_rules"],
  "gaps": [
    "<description of anything that could not be filled>"
  ]
}
```

## Quality bar before writing

Before writing any context file, verify:

- [ ] System purpose is a single sentence in business terms (no technical jargon).
- [ ] Every entity entry has a business meaning, not a schema description.
- [ ] Every business rule is stated as a constraint or invariant, not an implementation step.
- [ ] Every unknown is marked `[NEEDS INPUT: ...]`, never left blank.
- [ ] No file exceeds its token budget (glossary: 500, service: 600).
- [ ] No business rules were invented — every rule came from harvest or interview.
